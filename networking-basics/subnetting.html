✂️ Subnetting — Complete Guide

Subnetting = Dividing a large network into smaller, manageable subnetworks. Essential for IP address efficiency, security, and routing.

🎯 Why Subnetting Matters

  • ✅ Conserves IP addresses — Don't waste scarce IPv4 addresses
  • ✅ Improves security — Isolate departments (HR, Finance, Guest)
  • ✅ Reduces broadcast traffic — Less noise on each subnet
  • ✅ Simplifies troubleshooting — Smaller failure domains
💡 Real-world: A company with 200 devices doesn't need a /16 (65,534 addresses). Subnetting gives them a /24 (254 addresses) — efficient and secure.

📐 Understanding the Subnet Formula (Step by Step)

Formula 1: Total IP Addresses

Total IPs = 2 ^ (32 - CIDR_prefix)

Where ^ means "raised to the power of" (exponent).

Example: /24 network
Total IPs = 2 ^ (32 - 24) = 2 ^ 8 = 256 addresses
Example: /26 network
Total IPs = 2 ^ (32 - 26) = 2 ^ 6 = 64 addresses
💡 Read as: "2 raised to the power of (32 minus the CIDR prefix)"
Not "2 times (32 - prefix)" — the ^ means exponent/power!

Formula 2: Usable IP Addresses

Usable IPs = Total IPs - 2

Why subtract 2? Every subnet reserves:

  • Network address (all host bits 0) — cannot assign to devices
  • Broadcast address (all host bits 1) — used to reach all devices
Example: /24 network
Usable = 256 - 2 = 254 devices (192.168.1.1 → 192.168.1.254)

Formula 3: Block Size (Quick Mental Math)

Block size = 256 - last_non_zero_octet_of_mask

This works for /24 to /32 (where mask changes in the last octet).

Example: Mask 255.255.255.192 (/26)
Block = 256 - 192 = 64
Subnets: 0, 64, 128, 192
Example: Mask 255.255.255.224 (/27)
Block = 256 - 224 = 32
Subnets: 0, 32, 64, 96, 128, 160, 192, 224
📌 Remember: 2^something means "2 multiplied by itself that many times"
2^8 = 2×2×2×2×2×2×2×2 = 256
2^6 = 2×2×2×2×2×2 = 64

🎯 Finding Network Address, Broadcast, First & Last Host

Example: 192.168.1.45 /24

  • Network address: 192.168.1.0 (all host bits 0)
  • First usable host: 192.168.1.1
  • Last usable host: 192.168.1.254
  • Broadcast address: 192.168.1.255 (all host bits 1)

Example: 10.0.0.100 /26 (mask 255.255.255.192)

  • Blocks of 64 addresses (0-63, 64-127, 128-191, 192-255)
  • Network: 10.0.0.64
  • First host: 10.0.0.65
  • Last host: 10.0.0.126
  • Broadcast: 10.0.0.127
💡 Quick trick: Find the block size = 256 - octet value. For /26 (255.255.255.192), block = 256 - 192 = 64.

📝 Practice Questions (Test Yourself)

Q1: What is the subnet mask for /20?

Answer: 255.255.240.0 (16 + 4 = 20 bits, so 11111111.11111111.11110000.00000000)

Q2: 192.168.5.35/28 — What is the network address?

Answer: 192.168.5.32 (block size 16: 0,16,32,48 → 32-47 range)

Q3: How many usable IPs in a /29?

Answer: 2^(32-29)=8 total, minus 2 = 6 usable addresses.

Q4: What CIDR gives subnet mask 255.255.254.0?

Answer: /23 (255.255.254.0 = 23 network bits)

Q5: A /26 subnet — what are the valid host ranges in 10.0.0.0/26?

Answer: 10.0.0.1 - 10.0.0.62, 10.0.0.65 - 10.0.0.126, 10.0.0.129 - 10.0.0.190, 10.0.0.193 - 10.0.0.254

🏗️ Strategic Network Design: How to Plan Subnets for Real Companies

Stop guessing — learn the systematic method used by network architects.

📊 Step 1: Gather Requirements

Ask these questions before touching a calculator:

  • 📌 How many devices NOW? (workstations, servers, printers, IoT)
  • 📈 Growth rate? (20% per year? Double in 2 years?)
  • 🔒 Security zones? (HR separate from Guest? Finance isolated?)
  • 🌍 Physical locations? (One building? Multiple cities?)
  • 💰 Available IP space? (Do you have a /24, /20, or /16?)

🧮 Step 2: The "Powers of 2" Method (Find Smallest Usable Mask)

💡 Formula: 2^(host bits) - 2 ≥ Required hosts
Host bits = 32 - CIDR prefix

Example: Department needs 50 hosts

  • Try /26: 2^(6) - 2 = 64 - 2 = 62 hosts ✅ (fits, but 12 wasted)
  • Try /27: 2^(5) - 2 = 32 - 2 = 30 hosts ❌ (too small)
  • Winner: /26 (62 usable, room for 20% growth)

🗺️ Step 3: Hierarchical Addressing (The Professional Way)

Instead of random subnets, build a hierarchy:

📐 Step 4: The "Biggest-First" Allocation Rule

Always assign subnets to the largest requirements first. This prevents fragmentation.

⚖️ Step 5: Right-Sizing vs. Future Growth

  • ✅ Conservative: Exact fit + 10% — minimal waste
  • ✅ Aggressive: Exact fit — risky if growth happens
  • ✅ Liberal: Double the size — 50% waste but safe
⚠️ Real-world advice: Always leave 20-30% free in each subnet. Running out of IPs is a painful redesign.

🎨 Network Design Patterns (Proven Solutions to Common Problems)

Design patterns = reusable templates that save time and prevent mistakes.

📌 Pattern 1: Route Summarization (Aggregation)

Problem: Routers have too many routes, memory fills up, convergence slows.

Solution: Advertise one summary route instead of many detailed routes.

Without summarization (256 routes): 192.168.0.0/24, 192.168.1.0/24, 192.168.2.0/24 ... 192.168.255.0/24 With summarization (1 route): 192.168.0.0/16 ← One route covers ALL 256 subnets

How to find summary route:

  • Find common bits in all networks
  • 192.168.x.x has first 16 bits identical → /16 summary
  • 192.168.0.0/24 + 192.168.1.0/24 → summarize to 192.168.0.0/23
💡 Exam trick: To summarize 10.1.0.0/24, 10.1.1.0/24, 10.1.2.0/24, 10.1.3.0/24 → look for common first 22 bits → 10.1.0.0/22

📌 Pattern 2: VLSM Templates (Standardized Sizes)

Problem: Every subnet is different, hard to document and troubleshoot.

Solution: Create a template with fixed sizes for each use case.

📌 Pattern 3: Hierarchical Addressing (The Scalability Pattern)

Problem: Flat networks don't scale — routing tables explode.

Solution: Assign addresses by geography or function.

Global company using 10.0.0.0/8 as private space: Level 1: Continent (10.0.0.0/12) │ ├── North America (10.0.0.0/13) │ ├── NY (10.0.0.0/18) │ │ ├── Engineering (10.0.0.0/24) │ │ └── Sales (10.0.1.0/24) │ └── CA (10.0.64.0/18) │ ├── Engineering (10.0.64.0/24) │ └── Sales (10.0.65.0/24) │ └── Europe (10.8.0.0/13) ├── London (10.8.0.0/18) └── Berlin (10.8.64.0/18)

Benefits: Route summarization at each level, easy to add new sites, quick troubleshooting.

📌 Pattern 4: Sparse Allocation (Leave Growth Gaps)

Problem: You run out of IPs in a subnet and have to redesign everything.

Solution: Leave gaps between subnets for future expansion.

❌ Dense allocation (no gaps):
HR: 10.0.0.0/24, IT: 10.0.1.0/24, Sales: 10.0.2.0/24 (no room for growth)

✅ Sparse allocation (gaps):
HR: 10.0.0.0/24, (gap .1.0/24), IT: 10.0.2.0/24, (gap .3.0/24), Sales: 10.0.4.0/24
Gaps allow you to expand HR into 10.0.1.0/24 without redesigning IT or Sales!

📌 Pattern 5: Supernetting (CIDR Aggregation)

Problem: You have multiple small networks that need to be advertised as one.

Solution: Combine them into a larger block (opposite of subnetting).

  • Two /25s combine to one /24: 192.168.0.0/25 + 192.168.0.128/25 → 192.168.0.0/24
  • Four /24s combine to one /22: 10.1.0.0/24 + 10.1.1.0/24 + 10.1.2.0/24 + 10.1.3.0/24 → 10.1.0.0/22
⚠️ Supernetting only works when networks are contiguous (in a row, no gaps).

📌 Pattern 6: Overlay Networks (Virtual Topologies)

Problem: You need to connect subnets across different physical locations or incompatible networks.

Solution: Create a virtual network on top of physical infrastructure.

  • VPNs: Secure tunnel between two subnets over internet
  • VXLAN: Extend Layer 2 across Layer 3 networks (data centers)
  • GRE/ IPIP: Generic encapsulation for custom routing

📌 Pattern 7: IP Conservation (For IPv4 Scarcity)

Problem: IPv4 addresses are expensive and limited.

Solution: Use these patterns to save addresses:

  • /31 for point-to-point: Modern routers support /31 (2 hosts, no waste instead of /30's 50% waste)
  • NAT (Network Address Translation): Many private IPs share one public IP
  • RFC 6598 (CGNAT): 100.64.0.0/10 for carrier-grade NAT
  • IPv6 dual-stack: Run IPv6 alongside IPv4

🤔 Decision Tree: Which Design Pattern Should You Use?

Start here: │ ├── Need to save routing table space? │ └── YES → Use Route Summarization / Supernetting │ ├── Need consistent, repeatable designs? │ └── YES → Use VLSM Templates │ ├── Need to scale to multiple locations? │ └── YES → Use Hierarchical Addressing │ ├── Worried about running out of space later? │ └── YES → Use Sparse Allocation │ ├── Need to connect disjointed networks? │ └── YES → Use Overlay Networks (VPN, VXLAN) │ ├── Running out of IPv4 addresses? │ └── YES → Use IP Conservation (NAT, /31)

🎓 Key Takeaways: Design Patterns

  • ✅ Summarization = One route instead of many (smaller routing tables)
  • ✅ VLSM templates = Standard sizes for common scenarios
  • ✅ Hierarchical = Addresses reflect physical/logical topology
  • ✅ Sparse allocation = Leave gaps for future growth
  • ✅ Supernetting = Combine multiple networks into one
  • ✅ Overlays = Virtual networks on top of physical
  • ✅ Conservation = Strategies to save IPv4 addresses
