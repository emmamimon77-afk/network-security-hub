🧱 Zone-Based Firewall (ZBF) — Complete Guide

Zone-Based Firewall (ZBF) = Cisco's modern firewall model. Instead of per-interface ACLs, you group interfaces into zones and define policies between zones. More logical, scalable, and stateful.

🎯 Why Zone-Based Firewall Matters

  • ✅ Logical organization — Group interfaces by trust level (Inside, Outside, DMZ)
  • ✅ Stateful inspection — Tracks connections like iptables (not just packet-by-packet)
  • ✅ Simpler rules — One policy between zones instead of ACLs on every interface
  • ✅ Scalable — Add new interfaces to a zone, policy applies automatically
  • ✅ CCNP/CCIE required — Essential for advanced Cisco security exams
💡 Real-world: Enterprise firewalls (Cisco ASA, FTD) use zone-based logic. ZBF brings this power to IOS routers.

📊 ZBF vs Classic ACL (The Evolution)

FeatureClassic ACLZone-Based Firewall
Direction Inbound/outbound per interface Between zones (zone-pair)
Statefulness Stateless (evaluates each packet) Stateful (tracks connections)
Complexity High (many ACLs on many interfaces) Lower (one policy per zone-pair)
Scalability Poor (add interface = add ACLs) Excellent (add interface to zone)
Application awareness Layer 3/4 only Layer 7 (NBAR, can inspect HTTP, FTP, etc.)

📚 Core ZBF Concepts (The 3 Pillars)

1. Zones

A zone is a logical group of interfaces with the same security level.

zone security INSIDE
zone security OUTSIDE
zone security DMZ

2. Zone-Pairs

A zone-pair defines traffic direction between two zones (source → destination).

zone-pair security FROM-INSIDE-TO-OUTSIDE source INSIDE destination OUTSIDE

3. Policies

A policy defines WHAT traffic is allowed and HOW it's inspected. Attached to a zone-pair.

policy-map type inspect INSIDE-TO-OUTSIDE-POLICY
class type inspect ALLOWED-TRAFFIC
inspect
class class-default
drop

⚙️ ZBF Configuration (Step-by-Step)

Step 1: Define Zones

zone security INSIDE
zone security OUTSIDE
zone security DMZ

Step 2: Assign Interfaces to Zones

interface GigabitEthernet0/0
zone-member security INSIDE
!
interface GigabitEthernet0/1
zone-member security OUTSIDE
!
interface GigabitEthernet0/2
zone-member security DMZ

Step 3: Define Traffic Classes (What to Match)

! Allow HTTP, HTTPS, DNS, ICMP
class-map type inspect match-any ALLOWED-TRAFFIC
match protocol http
match protocol https
match protocol dns
match protocol icmp
!
! Allow SSH from INSIDE to DMZ
class-map type inspect SSH-TO-DMZ
match protocol ssh
match access-group name SSH-SOURCE
!
ip access-list extended SSH-SOURCE
permit ip 192.168.1.0 0.0.0.255 any

Step 4: Create Policy-Map (What to Do)

policy-map type inspect INSIDE-TO-OUTSIDE-POLICY
class type inspect ALLOWED-TRAFFIC
inspect
class class-default
drop log

Step 5: Create Zone-Pair and Apply Policy

zone-pair security FROM-INSIDE-TO-OUTSIDE source INSIDE destination OUTSIDE
service-policy type inspect INSIDE-TO-OUTSIDE-POLICY
💡 Result: INSIDE zone can initiate HTTP, HTTPS, DNS to OUTSIDE. OUTSIDE cannot initiate anything to INSIDE (stateful).

🏗️ Real Example: Web Server DMZ

Scenario:

  • INSIDE zone: Corporate LAN (192.168.1.0/24) — employees
  • DMZ zone: Public web server (10.0.0.5)
  • OUTSIDE zone: Internet

Requirements:

  • INSIDE → OUTSIDE: HTTP, HTTPS, DNS (allow)
  • OUTSIDE → DMZ: HTTP, HTTPS only (allow)
  • INSIDE → DMZ: SSH (management) only

Configuration:

! Step 1: Zones
zone security INSIDE
zone security OUTSIDE
zone security DMZ

! Step 2: Assign interfaces
int g0/0 — zone-member security INSIDE
int g0/1 — zone-member security OUTSIDE
int g0/2 — zone-member security DMZ

! Step 3: Class-maps
class-map type inspect match-any WEB-TRAFFIC
match protocol http
match protocol https

class-map type inspect SSH-ONLY
match protocol ssh

! Step 4: Policy-maps
policy-map type inspect INSIDE-TO-OUTSIDE
class type inspect WEB-TRAFFIC
inspect
class class-default
drop

policy-map type inspect OUTSIDE-TO-DMZ
class type inspect WEB-TRAFFIC
inspect
class class-default
drop

policy-map type inspect INSIDE-TO-DMZ
class type inspect SSH-ONLY
inspect
class class-default
drop

! Step 5: Zone-pairs
zone-pair security I-TO-O source INSIDE destination OUTSIDE
service-policy type inspect INSIDE-TO-OUTSIDE

zone-pair security O-TO-DMZ source OUTSIDE destination DMZ
service-policy type inspect OUTSIDE-TO-DMZ

zone-pair security I-TO-DMZ source INSIDE destination DMZ
service-policy type inspect INSIDE-TO-DMZ

🛡️ ZBF Security Best Practices

  • ✅ Self zone: The router itself is in the "self" zone. Control access to it with zone-pair self
  • ✅ Drop all by default: Always include class class-default drop in policies
  • ✅ Log drops: Use drop log to see denied packets (debug zone-pair security)
  • ✅ Use parametric class-maps: Match by protocol, not just port (HTTP not just port 80)
  • ✅ Inspect, not pass: inspect is stateful, pass is stateless (avoid pass)
  • ✅ Never put two interfaces in same zone if traffic must be filtered — same zone = no inspection
⚠️ Critical: Interfaces in the same zone have no firewall policy between them — they route freely!

🔧 Troubleshooting ZBF

❌ Traffic not passing as expected?
Check:
1. Zone membership: show zone-pair security
2. Policy attached: show policy-map type inspect zone-pair
3. Hit counts: show policy-map type inspect zone-pair sessions
4. Debug drops: debug zone-pair security drops
❌ Internet works, but can't SSH to DMZ server?
Check:
- Is there a zone-pair from INSIDE to DMZ?
- Does the policy allow SSH? (class-map type inspect SSH-ONLY)
- Is the SSH policy inspecting or passing?

📝 Practice Questions

Q1: What is the difference between inspect and pass in a ZBF policy?

Answer: inspect is stateful (tracks connections, allows return traffic). pass is stateless (allows packet through without tracking). Always use inspect.

Q2: Two interfaces in the same zone. Is traffic inspected between them?

Answer: No. ZBF only inspects traffic between different zones. Same zone = no policy = routing only.

Q3: What command creates a zone-pair for traffic from INSIDE to OUTSIDE with policy named MY-POLICY?

Answer: zone-pair security MY-PAIR source INSIDE destination OUTSIDE
Then: service-policy type inspect MY-POLICY

Q4: Why is ZBF considered more scalable than traditional ACLs?

Answer: With ACLs, adding an interface requires editing ACLs on that interface. With ZBF, you just add the interface to a zone, and existing zone-pair policies apply automatically.

🧪 Lab Exercise: Build a Basic ZBF

Goal: Configure zone-based firewall on a Cisco router with INSIDE, OUTSIDE, and DMZ zones.

Step 1: Define zones & assign interfaces

zone security INSIDE
zone security OUTSIDE
zone security DMZ
int g0/0 — zone-member security INSIDE
int g0/1 — zone-member security OUTSIDE

class-map type inspect match-any WEB
match protocol http
match protocol https

class-map type inspect match-all SSH-ONLY

Step 3: Create policies

policy-map type inspect INSIDE-TO-OUTSIDE
class type inspect WEB
inspect
class class-default
drop log
!
policy-map type inspect OUTSIDE-TO-DMZ
class type inspect WEB
inspect
class class-default
drop

Step 4: Apply to zone-pairs

zone-pair security I-TO-O source INSIDE destination OUTSIDE
service-policy type inspect INSIDE-TO-OUTSIDE
!
zone-pair security O-TO-DMZ source OUTSIDE destination DMZ
service-policy type inspect OUTSIDE-TO-DMZ

🎓 Key Takeaways (ZBF Mastery)

  • ✅ Zones group interfaces by trust level (INSIDE, OUTSIDE, DMZ)
  • ✅ Zone-pairs define direction (source → destination)
  • ✅ Policies define what's allowed and how to inspect it
  • ✅ Always use inspect for stateful firewalling — never pass
  • ✅ Interfaces in the same zone are NOT inspected — they route freely
  • ✅ ZBF is scalable and stateful — the modern way to firewall on Cisco IOS
