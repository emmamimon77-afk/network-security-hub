🛠️ iptables — Complete Guide

iptables = The built-in firewall on Linux systems. Filters packets, performs NAT, and protects servers. Every security professional must master it.

🎯 Why iptables Matters

  • ✅ Linux standard — Every Linux server has it (cloud, on-prem, containers)
  • ✅ High performance — Packet filtering in kernel space (very fast)
  • ✅ Granular control — Filter by IP, port, protocol, connection state
  • ✅ NAT & forwarding — Share internet, build routers, create VPN gateways
  • ✅ Cloud & devops — Essential for AWS, Azure, Kubernetes security groups
💡 Real-world: Every time you spin up a Linux server (on AWS, DigitalOcean, or on-prem), iptables (or its modern successor nftables) is protecting it.

📊 The 3 Default Chains (Where Packets Go)

┌─────────────────────────────────────────────────────────────────┐ │ INPUT CHAIN │ │ Packets destined FOR the firewall itself (ssh, web, etc.) │ └─────────────────────────────────────────────────────────────────┘ ▲ │ ┌─────────────────────────────────────────────────────────────────┐ │ FORWARD CHAIN │ │ Packets that pass THROUGH the firewall (router mode) │ └─────────────────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────────────┐ │ OUTPUT CHAIN │ │ Packets generated BY the firewall itself │ └─────────────────────────────────────────────────────────────────┘
ChainDirectionTypical Rules
INPUT (most important) Traffic to the server Allow SSH (22), HTTP (80), HTTPS (443). Block everything else.
FORWARD Traffic through the server (router mode) Used when Linux is a router or gateway.
OUTPUT Traffic from the server Usually allow all (server can talk out), but can restrict.

📐 iptables Rule Anatomy (Match + Target)

iptables -A INPUT -p tcp --dport 22 -j ACCEPT
  • -A INPUT → Append rule to INPUT chain
  • -p tcp → Match protocol (tcp, udp, icmp)
  • --dport 22 → Match destination port (SSH)
  • -j ACCEPT → Target (ACCEPT, DROP, REJECT, LOG)

🔗 Connection Tracking (Stateful Firewall)

iptables can track connection states — this is what makes it a true stateful firewall.

iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
  • NEW: First packet of a new connection
  • ESTABLISHED: Part of an existing connection (reply traffic)
  • RELATED: Related to an existing connection (e.g., FTP data channel)
  • INVALID: Malformed or bogus packets
💡 Best practice: Allow ESTABLISHED,RELATED at the top of INPUT. This lets your server respond to traffic you initiated, without opening unnecessary ports.

⚙️ Essential iptables Commands

Viewing Rules

iptables -L -v -n # List all rules (verbose, numeric)
iptables -L INPUT # List only INPUT chain
iptables -S # Show rules as commands (useful for backup)

Adding Rules

iptables -A INPUT -p tcp --dport 22 -j ACCEPT # Append rule
iptables -I INPUT 1 -p tcp --dport 80 -j ACCEPT # Insert at position 1
iptables -A INPUT -j DROP # Default deny (last rule!)

Deleting Rules

iptables -D INPUT -p tcp --dport 22 -j ACCEPT # Delete by specification
iptables -D INPUT 3 # Delete by rule number
iptables -F # Flush ALL rules (careful!)

Saving & Restoring (Rules reset on reboot)

# Save current rules to file
iptables-save > /etc/iptables/rules.v4

# Restore from file
iptables-restore < /etc/iptables/rules.v4
⚠️ Critical: iptables rules are NOT persistent across reboots. Always save them after changes!

🌐 Practical Examples (Real Server Security)

Example 1: Basic Web Server Firewall

# Flush existing rules
iptables -F

# Default policies (drop everything unless allowed)
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Allow established connections
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow SSH, HTTP, HTTPS
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Log dropped packets (optional, for debugging)
iptables -A INPUT -j LOG --log-prefix "Dropped: "

Example 2: Rate Limiting SSH (Prevent Brute Force)

# Allow 4 SSH attempts per minute from same IP
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --set
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --update --seconds 60 --hitcount 4 -j DROP
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

Example 3: Block Specific IP (Blacklist)

iptables -A INPUT -s 203.0.113.5 -j DROP

Example 4: Allow Only Specific Subnet (Management)

# Only allow SSH from corporate network
iptables -A INPUT -p tcp --dport 22 -s 192.168.0.0/16 -j ACCEPT

🔄 NAT with iptables (Internet Sharing / Port Forwarding)

MASQUERADE (Share Internet — Home Router Style)

# Enable IP forwarding
echo 1 > /proc/sys/net/ipv4/ip_forward

# Masquerade traffic going out eth0 (internet)
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

# Allow forwarding (basic)
iptables -A FORWARD -i eth1 -o eth0 -j ACCEPT
iptables -A FORWARD -i eth0 -o eth1 -m state --state ESTABLISHED,RELATED -j ACCEPT

Port Forwarding (DNAT)

# Forward port 8080 on firewall to port 80 on internal server 192.168.1.100
iptables -t nat -A PREROUTING -p tcp --dport 8080 -j DNAT --to-destination 192.168.1.100:80
💡 Use case: A Linux server with two network interfaces can act as a router/firewall for an entire office or home lab.

🛡️ iptables Security Best Practices

  • ✅ Default deny policy: iptables -P INPUT DROP — block everything first
  • ✅ Allow ESTABLISHED,RELATED first: Prevents breaking active connections
  • ✅ Always allow loopback: -A INPUT -i lo -j ACCEPT (many services need it)
  • ✅ Rate limit SSH: Prevent brute force attacks
  • ✅ Log dropped packets: Helps debug and detect attacks
  • ✅ Save rules regularly: iptables-save > /etc/iptables/rules.v4
  • ✅ Test with iptables -L -v -n before applying to production

📝 Practice Questions

Q1: What command blocks all incoming SSH (port 22) except from 10.0.0.0/8?

Answer: iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/8 -j ACCEPT
Then: iptables -A INPUT -p tcp --dport 22 -j DROP

Q2: Why is the -m state --state ESTABLISHED,RELATED rule important?

Answer: It allows your server to respond to traffic it initiated (like DNS responses, web replies) without opening inbound ports permanently. Essential for stateful firewalls.

Q3: Your iptables rules disappear after reboot. Why?

Answer: iptables rules are stored in memory only. Use iptables-save and iptables-restore with a startup script to make them persistent.

Q4: What's the difference between DROP and REJECT targets?

Answer: DROP silently discards packets. REJECT sends an error (ICMP port unreachable). REJECT helps clients know immediately, DROP is stealthier.

Q5: You need to forward port 8080 on your firewall to port 80 on internal server 192.168.1.10. What command do you use?

Answer: iptables -t nat -A PREROUTING -p tcp --dport 8080 -j DNAT --to-destination 192.168.1.10:80

🎓 Key Takeaways (iptables Mastery)

  • ✅ 3 default chains: INPUT (to server), FORWARD (through server), OUTPUT (from server)
  • ✅ Default deny is safest: iptables -P INPUT DROP
  • ✅ Stateful filtering: Allow ESTABLISHED,RELATED first
  • ✅ Rules processed in order: First match wins!
  • ✅ Always allow loopback (-i lo) — many services need it
  • ✅ Rules are not persistent: Save with iptables-save
  • ✅ Check rules before applying: iptables -L -v -n
