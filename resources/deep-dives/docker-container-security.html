🐳 Docker & Container Security — Deep Dive

Containers have revolutionized application deployment, but they introduce unique security challenges. This deep dive covers container isolation, image security, runtime protection, Kubernetes security contexts, and industry best practices.

💡 Target audience: DevOps engineers, security professionals, and anyone running containers in production.

🔒 Container Isolation — How Containers Stay Separate

Containers are not virtual machines. They share the host kernel but are isolated using two key Linux features:

  • Namespaces: Provide process isolation (PID, NET, MNT, UTS, IPC, USER, CGROUP).
  • Control Groups (Cgroups): Limit and meter resources (CPU, memory, disk I/O).

📦 Image Security — Building Secure Containers

The supply chain starts with the image. Compromised images lead to compromised containers.

# Dockerfile Best Practices # 1. Use specific, minimal base images FROM alpine:3.18 # NOT FROM ubuntu:latest # 2. Create a non-root user RUN addgroup -g 1001 -S appgroup && \ adduser -u 1001 -S appuser -G appgroup # 3. Run as non-root user USER appuser # 4. Pin dependency versions RUN pip install flask==2.2.5 # 5. Use multi-stage builds to reduce size FROM golang:1.19-alpine AS builder # ... build steps ... FROM alpine:3.18 COPY --from=builder /app/binary .
📋 Security scanning with Trivy:
trivy image nginx:latest

🛡️ Runtime Security — Running Containers Securely

Prevent containers from being the weakest link in your infrastructure.

# 1. Drop all capabilities, then add only needed ones docker run --cap-drop=ALL --cap-add=NET_RAW nginx # 2. Set resource limits (prevent DoS) docker run --memory="512m" --cpus="1.0" nginx # 3. Make root filesystem read-only (prevents writes to container) docker run --read-only nginx # 4. Use Docker Bench Security to audit your host docker run --net host --pid host --userns host --cap-add audit_control \ -e DOCKER_CONTENT_TRUST=$DOCKER_CONTENT_TRUST \ -v /var/lib:/var/lib \ -v /var/run/docker.sock:/var/run/docker.sock \ -v /usr/lib/systemd:/usr/lib/systemd \ -v /etc/docker:/etc/docker \ --label docker_bench_security \ docker/docker-bench-security
💡 Pro tip: Never run containers with the `--privileged` flag in production. It disables all security features.

☸️ Kubernetes Security Contexts — Pod-Level Security

In Kubernetes, a `securityContext` defines privilege and access control settings for a Pod or Container.

apiVersion: v1 kind: Pod metadata: name: secure-pod spec: securityContext: runAsNonRoot: true runAsUser: 1001 fsGroup: 2001 seccompProfile: type: RuntimeDefault containers: - name: app image: nginx securityContext: allowPrivilegeEscalation: false capabilities: drop: ["ALL"] add: ["NET_ADMIN"]

🔗 Supply Chain Security — SBOM, Provenance, Signing

  • SBOM (Software Bill of Materials): A complete inventory of all components in your container.
  • Provenance: Verifiable attestation of how and where the image was built.
  • Image Signing (Cosign): Cryptographically sign images to ensure authenticity.
# Generate SBOM with Syft syft nginx:latest -o spdx-json > sbom.json # Sign an image with Cosign cosign generate-key-pair cosign sign --key cosign.key nginx:latest cosign verify --key cosign.pub nginx:latest

👀 Monitoring at Runtime — Falco

Falco is the open-source standard for runtime security, detecting anomalous behavior.

# Install Falco curl -fsSL https://falco.org/repo/falcosecurity-packages.asc | sudo apt-key add - echo "deb https://download.falco.org/packages/deb stable main" | sudo tee /etc/apt/sources.list.d/falcosecurity.list sudo apt update && sudo apt install -y falco # Run Falco sudo falco

📝 Practice Exercises

Exercise 1: Create a secure Dockerfile for a Python app that runs as a non-root user.
Solution:
FROM python:3.10-slim
RUN useradd -m appuser
USER appuser
COPY --chown=appuser:appuser . /app
CMD ["python", "/app/app.py"]
Exercise 2: Scan a container image for vulnerabilities using Trivy.
Solution:
trivy image nginx:latest
Exercise 3: Run an nginx container with read-only root filesystem and resource limits.
Solution:
docker run -d --read-only --memory="256m" --cpus="0.5" nginx
Exercise 4: What is the purpose of a Kubernetes `securityContext`?
Answer: It defines privilege and access control settings for a Pod or Container, such as running as a non-root user or dropping all Linux capabilities.

🎓 Key Takeaways (Container Security)

  • ✅ Isolation: Namespaces and cgroups are the foundation of container security.
  • ✅ Images: Use minimal base images, pin versions, and run as non-root.
  • ✅ Runtime: Drop capabilities, set resource limits, and never use `--privileged`.
  • ✅ Kubernetes: Use `securityContext` to enforce pod-level security.
  • ✅ Supply chain: Generate SBOMs, sign images, and verify provenance.
  • ✅ Monitoring: Use Falco for runtime anomaly detection.
