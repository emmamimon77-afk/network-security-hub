🐧 Linux Security Hardening — Deep Dive

Transform your Linux system into a fortress. This deep dive covers Mandatory Access Control (SELinux/AppArmor), kernel hardening, auditd monitoring, system security benchmarks, and practical hardening commands you can apply today.

💡 Target audience: Linux administrators, security professionals, and anyone running a production Linux server.

🎯 Why Linux Security Hardening Matters

  • ✅ Linux powers 90%+ of cloud servers — from AWS to Azure to Google Cloud
  • ✅ Default installations are NOT secure — they're configured for convenience
  • ✅ Compliance requirements — PCI-DSS, HIPAA, GDPR demand hardening
  • ✅ Defense in depth — multiple layers protect against zero-day exploits
⚠️ Reality check: Most breaches exploit misconfigurations, not zero-days. Hardening eliminates low-hanging fruit.

📊 Step 1: Audit Your System — Know What You're Hardening

Essential Audit Commands:

# Check OS version
cat /etc/os-release

# List all running services
systemctl list-units --type=service --state=running

# Check open ports and listening services
ss -tulpn

# List all installed packages
dpkg -l | wc -l (Debian/Ubuntu)
rpm -qa | wc -l (RHEL/CentOS)

# Check for unnecessary users
cat /etc/passwd | grep -E "/bin/bash|/bin/sh"

# Check sudoers configuration
cat /etc/sudoers
cat /etc/sudoers.d/*
📋 Create a baseline report:
mkdir ~/security-audit-$(date +%Y%m%d)
ss -tulpn > ~/security-audit-$(date +%Y%m%d)/open-ports.txt
systemctl list-units --type=service --state=running > ~/security-audit-$(date +%Y%m%d)/running-services.txt
cat /etc/passwd > ~/security-audit-$(date +%Y%m%d)/users.txt

🔒 Step 2: Mandatory Access Control — SELinux vs AppArmor

MAC systems restrict even root processes. They're the difference between a compromised service and a fully compromised server.

FeatureAppArmor (Ubuntu/Debian)SELinux (RHEL/CentOS/Fedora)
Default on Ubuntu, Debian, SUSE RHEL, CentOS, Fedora, Amazon Linux
Configuration Per-profile (easier) Policy-based (more complex)
Learning curve Gentle Steep

AppArmor Commands (Ubuntu/Debian/Pop!_OS — Your Laptop!)

# Check AppArmor status
sudo aa-status

# List loaded profiles
sudo apparmor_status

# Put a profile into complain mode (log violations without blocking)
sudo aa-complain /etc/apparmor.d/usr.sbin.nginx

# Put a profile into enforce mode (block violations)
sudo aa-enforce /etc/apparmor.d/usr.sbin.nginx

# Reload all profiles
sudo systemctl reload apparmor

# Disable AppArmor (not recommended!)
sudo systemctl stop apparmor
sudo systemctl disable apparmor
💡 Pro tip: On your new Ubuntu laptop, run sudo aa-status to see which processes are confined. You'll likely see Docker, Snap packages, and system services.

🔧 Step 3: Kernel Hardening — /etc/sysctl.conf

The Linux kernel has hundreds of security knobs. These are essential for production servers.

Critical Kernel Security Parameters:

# /etc/sysctl.d/99-security.conf

# IP Spoofing protection
net.ipv4.conf.all.rp_filter=1
net.ipv4.conf.default.rp_filter=1

# Ignore ICMP redirects (prevent man-in-the-middle)
net.ipv4.conf.all.accept_redirects=0
net.ipv4.conf.default.accept_redirects=0

# Ignore source route packets
net.ipv4.conf.all.accept_source_route=0
net.ipv4.conf.default.accept_source_route=0

# Enable TCP SYN cookie protection (mitigates SYN flood)
net.ipv4.tcp_syncookies=1

# Log martian packets (spoofed, source-routed, etc.)
net.ipv4.conf.all.log_martians=1

# Disable IPv6 if not needed
net.ipv6.conf.all.disable_ipv6=1
net.ipv6.conf.default.disable_ipv6=1

# Restrict kernel pointer access (mitigates certain exploits)
kernel.kptr_restrict=2

# Restrict dmesg access (prevents kernel info leaks)
kernel.dmesg_restrict=1
📋 Apply kernel settings:
sudo sysctl -p /etc/sysctl.d/99-security.conf
sudo sysctl --system (apply all)

📝 Step 4: auditd — Kernel-Level Auditing

auditd watches file access, system calls, and user actions — critical for intrusion detection and compliance.

Install and Configure auditd:

# Install auditd
sudo apt install auditd audispd-plugins -y

# Add audit rules (/etc/audit/rules.d/audit.rules)
# Monitor /etc/passwd (user changes)
-w /etc/passwd -p wa -k user_changes
# Monitor /etc/shadow (password changes)
-w /etc/shadow -p wa -k password_changes
# Monitor /etc/sudoers (sudo privileges)
-w /etc/sudoers -p wa -k sudo_changes
# Monitor SSH configuration
-w /etc/ssh/sshd_config -p wa -k ssh_changes
# Monitor critical binaries
-w /bin/systemctl -p x -k systemctl_exec
-w /bin/rm -p x -k rm_exec
# Monitor failed file access
-a always,exit -S openat -F success=0 -k failed_access
📋 Working with audit logs:
sudo ausearch -k user_changes
sudo aureport --summary
sudo aureport --failed
sudo tail -f /var/log/audit/audit.log

🔑 Step 5: SSH Hardening — Protect Remote Access

# /etc/ssh/sshd_config — CRITICAL CHANGES

# Disable root login (attackers target root!)
PermitRootLogin no

# Use key-based authentication only
PasswordAuthentication no
PubkeyAuthentication yes

# Change default port (reduces bot scans)
Port 2222

# Limit login attempts
MaxAuthTries 3
MaxSessions 5

# Disable empty passwords
PermitEmptyPasswords no

# Restrict users who can SSH (if applicable)
AllowUsers alice bob

# Configure idle timeout
ClientAliveInterval 300
ClientAliveCountMax 2

# Apply changes
sudo systemctl restart sshd
⚠️ WARNING: Before changing SSH config, keep another terminal open with a root session. If you lock yourself out, you'll need physical access to recover!

🤖 Step 6: fail2ban — Automatic Attacker Blocking

# Install fail2ban
sudo apt install fail2ban -y

# Create local configuration
sudo cp /etc/fail2ban/jail.conf /etc/fail2ban/jail.local

# Configure SSH protection
[sshd]
enabled = true
port = ssh
maxretry = 3
bantime = 3600
findtime = 600

# Enable and start
sudo systemctl enable fail2ban --now

# Check banned IPs
sudo fail2ban-client status sshd

# Unban an IP
sudo fail2ban-client set sshd unbanip 203.0.113.5

📋 Step 7: CIS Benchmarks — Industry Standard

The CIS (Center for Internet Security) Benchmarks are the gold standard for system hardening. Use automated tools to check compliance.

# Install Lynis (security auditing tool)
sudo apt install lynis -y

# Run a system audit
sudo lynis audit system

# Generate report
sudo lynis show details

# Install oscap (OpenSCAP) for CIS benchmarks
sudo apt install openscap-scanner -y

# Run CIS benchmark scan (Ubuntu)
sudo oscap xccdf eval --profile xccdf_org.ssgproject.content_profile_cis --results scan-results.xml /usr/share/xml/scap/ssg/content/ssg-ubuntu2204-ds.xml
💡 Lynis tip: Run sudo lynis audit system on your laptop. It will show you exactly what hardening steps are missing on your own system!

🛡️ Step 8: Additional Hardening Measures

Automatic Security Updates:

# Enable unattended upgrades
sudo apt install unattended-upgrades -y
sudo dpkg-reconfigure --priority=low unattended-upgrades

Disable Unused Network Services:

# Remove unnecessary packages
sudo apt remove --purge telnet rsh-server rsh-client

# Disable IPv6 if not needed
echo 'net.ipv6.conf.all.disable_ipv6=1' | sudo tee -a /etc/sysctl.conf

File Integrity Monitoring (AIDE):

# Install AIDE
sudo apt install aide -y

# Initialize database (baseline)
sudo aideinit
sudo cp /var/lib/aide/aide.db.new /var/lib/aide/aide.db

# Run integrity check
sudo aide --check

🎓 Key Takeaways (Linux Security Hardening)

  • ✅ Audit first: Know your baseline with lynis or manual checks
  • ✅ AppArmor/SELinux — Mandatory Access Control limits damage from exploits
  • ✅ Kernel hardening — sysctl parameters block common attacks (IP spoofing, SYN flood)
  • ✅ auditd — Kernel-level logging for compliance and intrusion detection
  • ✅ SSH hardening — Disable root login, use keys, change port
  • ✅ fail2ban — Automatic brute-force protection
  • ✅ CIS Benchmarks — Industry standard; use Lynis to check compliance
  • ✅ Regular updates — Enable unattended-upgrades for security patches
