🌐 BGP Deep Dive — Border Gateway Protocol

Border Gateway Protocol (BGP) is the routing protocol of the internet. It connects autonomous systems (ASes) — ISPs, large enterprises, cloud providers — exchanging routing information across the globe. This deep dive covers BGP fundamentals, path attributes, route selection, iBGP/eBGP, route reflectors, communities, and BGP security (RPKI).

💡 Target audience: Network engineers, ISP staff, and anyone who wants to understand how the internet routes traffic.

🎯 BGP Basics — Autonomous Systems

An Autonomous System (AS) is a collection of IP networks under a single administrative entity (ISP, large company, university). Each AS has a unique 16-bit or 32-bit number (ASN).

  • Public ASN: Used on the internet (1-64495, 65536-65551, 131072-4199999999)
  • Private ASN: Used internally (64512-65535)
  • Example: Google's ASN is 15169, Cloudflare is 13335, your home router uses no ASN.

eBGP vs iBGP

eBGP (External BGP): Between different ASes ┌──────────┐ eBGP ┌──────────┐ │ AS 65001│◀──────▶│ AS 65002│ └──────────┘ └──────────┘ iBGP (Internal BGP): Within the same AS ┌──────────┐ iBGP ┌──────────┐ │ Router A │◀──────▶│ Router B │ │ (AS 65001) │ (AS 65001)│ └──────────┘ └──────────┘
  • eBGP: Used between different ASes. AD (Administrative Distance) = 20. Requires directly connected or multihop.
  • iBGP: Used within the same AS. AD = 200. Does not require direct connection. iBGP routers must be fully meshed or use route reflectors.

📊 BGP Path Attributes — How BGP Chooses the Best Path

BGP selects the best path based on a list of attributes, processed in order (decision process).

AttributeTypeDescriptionPreference
Weight (Cisco) Cisco proprietary Highest local preference (router-specific) Higher is better
Local Preference Well-known discretionary Exchanged within iBGP (higher = more preferred). Used to influence exit path Higher is better
AS Path Well-known mandatory List of ASNs the route has traversed (loop prevention) Shorter is better
Origin Well-known mandatory How route was injected: i (IGP), e (EGP), ? (incomplete/redistributed) i > e > ?
MED (Multi-Exit Discriminator) Optional non-transitive Influences inbound path from neighbor AS (lower = better) Lower is better
💡 Decision order: Weight (highest) → Local Pref (highest) → Originate locally → AS Path (shortest) → Origin (i > e > ?) → MED (lowest) → eBGP over iBGP → IGP metric → Router ID (lowest).
Many attributes exist; this is the Cisco decision process.

🔄 Scaling iBGP — Route Reflectors and Confederations

iBGP requires a full mesh of sessions between routers. For large ASes, this scales poorly (n(n-1)/2 sessions). Route reflectors and confederations reduce this.

Route Reflector (RR)

Without RR (full mesh, 6 sessions): R1────R2 │＼ ／│ │ X │ │／ ＼│ R3────R4 With RR (1 session each to RR, 3 sessions total): ┌────┐ │ RR │ └─┬──┘ ┌────┼────┐ R1 R2 R3 R4
! Configure route reflector on RR router bgp 65001 neighbor 10.0.0.2 route-reflector-client neighbor 10.0.0.3 route-reflector-client neighbor 10.0.0.4 route-reflector-client

Confederations (sub-ASes)

Split a large AS into smaller sub-ASes that communicate as eBGP within the confederation.

router bgp 65001 bgp confederation identifier 65100 bgp confederation peers 65002 65003 neighbor 10.0.1.2 remote-as 65002

🏷️ BGP Communities — Tagging Routes for Policy

Communities are optional transitive attributes that allow tagging routes with a 32-bit value (often written as AS:number). They simplify policy enforcement.

Well-known Communities:

  • NO_EXPORT (0xFFFFFF01) — Do not advertise to eBGP peers
  • NO_ADVERTISE (0xFFFFFF02) — Do not advertise to any peer
  • LOCAL_AS (0xFFFFFF03) — Do not advertise outside the local AS
! Set community on a route route-map SET-COMMUNITY permit 10 set community 65001:100 additive ! router bgp 65001 neighbor 192.168.1.2 route-map SET-COMMUNITY out

🛡️ BGP Security — RPKI, Prefix Filtering, and BGPSEC

BGP was designed without built-in security, making it vulnerable to route leaks and hijacks. RPKI (Resource Public Key Infrastructure) adds cryptographic validation.

Common BGP Attacks:

  • Route hijack: Announcing a prefix you don't own (e.g., BGP hijacking of AWS routes in 2018).
  • Route leak: Leaking routes from one AS to another violating policies.
  • AS path spoofing: Forging AS path to manipulate path selection.

RPKI (Resource Public Key Infrastructure):

💡 RPKI uses ROAs (Route Origin Authorizations) — a cryptographically signed attestation that an AS is authorized to announce a specific prefix.

Validation result: Valid (ROA exists, matches), Invalid (ROA exists but mismatched), NotFound (no ROA).

Prefix Filtering Best Practices:

! Example: Limit received prefixes from neighbor router bgp 65001 neighbor 192.168.1.2 maximum-prefix 1000 80 restart 30 ! ! Filter routes from a specific AS ip as-path access-list 10 deny _65002_ route-map DENY-AS65002 permit 10 match as-path 10 set community no-advertise

🗺️ BGP Route Aggregation (Summarization)

Aggregating prefixes reduces global routing table size and hides internal flapping.

router bgp 65001 aggregate-address 192.168.0.0 255.255.0.0 summary-only aggregate-address 10.0.0.0 255.0.0.0 as-set

📊 BGP Decision Process Example

Scenario: Router receives three routes to 203.0.113.0/24:
- Route A: AS path 65002 65005, Local Pref 100, MED 50
- Route B: AS path 65003, Local Pref 150, MED 20
- Route C: AS path 65002 65005 65007, Local Pref 100, MED 30 (learned via eBGP with weight 200)

Decision:
1. Weight: Route C (200) > others → wins.
2. If no weight, highest Local Pref (B has 150).
3. Then shortest AS path (B has length 1).
4. Then lowest origin? B is i (IGP) → same as others.
5. Then lowest MED (B has 20).
6. Then eBGP over iBGP, then IGP metric, then lowest Router ID.

📝 Practice Exercises

Exercise 1: Determine which BGP path would be chosen given: Path1: AS_PATH 65001 65002 65003, Local Pref 200; Path2: AS_PATH 65004, Local Pref 150.
Answer: Path1 because Local Pref (200) is higher than Path2 (150). AS_PATH length is considered only after Local Pref.
Exercise 2: Why are full meshes required for iBGP?
Answer: iBGP routers do not advertise routes learned from one iBGP peer to another iBGP peer (to prevent loops). A full mesh ensures every router receives all routes directly.
Exercise 3: What does the BGP community NO_EXPORT do?
Answer: It prevents a route from being advertised to eBGP peers (keeps it within the local AS).
Exercise 4: What is RPKI and how does it improve BGP security?
Answer: RPKI uses cryptographic certificates (ROAs) to validate that an AS is authorized to advertise a prefix, mitigating route hijacks.

🎓 Key Takeaways (BGP)

  • ✅ BGP = routing protocol of the internet — connects ASes
  • ✅ eBGP (between ASes, AD 20) and iBGP (within AS, AD 200) are distinct
  • ✅ Path selection — Weight → Local Pref → AS Path → Origin → MED → eBGP over iBGP → IGP metric → Router ID
  • ✅ Route reflectors and confederations reduce iBGP full mesh requirements
  • ✅ Communities enable scalable routing policies (e.g., tagging routes for NO_EXPORT)
  • ✅ RPKI adds cryptographic validation to prevent route hijacks
  • ✅ Always filter incoming routes — limit prefixes, validate AS paths, block bogon announcements
