🔬 Linux Level 5: Advanced & Deep Dive

For security researchers, malware analysts, and kernel developers. These commands peer into the deepest layers of Linux — system calls, kernel events, and process internals.

💡 Prerequisite: Linux Levels 1-4. These commands require deep understanding of Linux internals. Test in a safe VM first!

🔍 strace — See Every System Call a Program Makes

strace intercepts and records system calls — the interface between a program and the Linux kernel. Essential for debugging, reverse engineering, and understanding malware behavior.

CommandSyntaxDescriptionExample & Explanation
strace (trace program) strace /path/to/program Run a program and trace all its system calls strace ls
→ Shows every system call `ls` makes: opening directories, reading files, writing output.
Sample output snippet:
openat(AT_FDCWD, ".", O_RDONLY|O_NONBLOCK|O_CLOEXEC|O_DIRECTORY) = 3
Translation: "ls opened the current directory (file descriptor 3) to read its contents."
strace -p (attach to running process) sudo strace -p [PID] Attach to an already running process sudo strace -p 1234
→ Attaches to process with PID 1234 and shows its system calls in real-time.
Use case: Investigate a running suspicious process without restarting it.
🔧 Malware Analysis with strace:
strace -f -e trace=file,network ./suspicious_binary
→ -f (follow child processes/forks), -e trace=file,network (only show file and network operations)
Result: See exactly which files the malware touches and what network connections it makes — without running it in a debugger!
📖 Understanding System Calls (Common ones you'll see):
• open() / openat() → Opening a file
• read() / write() → Reading from or writing to a file/socket
• socket() / connect() → Creating network connections
• fork() / clone() → Creating a new process
• execve() → Replacing the current process with a new program
• kill() → Sending a signal to another process

⚡ perf — Linux Profiler (Performance Analysis)

perf is the official Linux performance monitoring tool. It can sample CPU usage, count hardware events (cache misses, branch predictions), and profile kernel activity.

CommandSyntaxDescriptionExample & Explanation
perf top sudo perf top Real-time performance monitoring (similar to `top`, but for CPU functions) sudo perf top
→ Shows which functions are using the most CPU, including kernel functions.
Use case: Find out what's slowing down your server when regular `top` only shows process-level usage.
🔧 Record and analyze system-wide performance:
sudo perf record -a -g -- sleep 30
→ Records performance data for 30 seconds across the entire system (-a = all CPUs, -g = capture call graphs).
sudo perf report
→ Generates an interactive report showing where CPU time was spent.
Use case: Profiling a production server to find bottlenecks.

🪄 bpftrace / eBPF — Program the Kernel

eBPF (Extended Berkeley Packet Filter) allows you to run sandboxed programs inside the Linux kernel without changing kernel code. bpftrace is the front-end for writing one-liners and scripts.

CommandSyntaxDescriptionExample & Explanation
bpftrace (trace syscalls) sudo bpftrace -e 'kprobe:do_nanosleep { printf("sleep\n"); }' Trace when any program calls the `nanosleep` syscall sudo bpftrace -e 'kprobe:do_nanosleep { printf("sleep called by PID %d\n", pid); }'
→ Prints a message every time any program calls `sleep()`.
Use case: Detect if a hidden process is using timers to evade detection.
📖 eBPF — The Future of Linux Observability:
eBPF powers modern tools like Cilium (networking), Falco (security monitoring), and Pixie (Kubernetes debugging). It's safe because programs are verified before loading into the kernel.
Installation: `sudo apt install bpftrace -y` (Ubuntu/Debian)
⚠️ Warning: eBPF requires kernel 4.1+ (preferred 5.4+). Check with `uname -r`. Some cloud providers restrict eBPF features in their VMs.

🐛 gdb — The GNU Debugger

gdb allows you to see what's happening inside a program while it runs — set breakpoints, inspect memory, change variables, and analyze crashes. Essential for reverse engineering and debugging complex bugs.

CommandSyntaxDescriptionExample & Explanation
gdb (run program) gdb ./program Start debugging a program gdb ./program
→ Enters gdb interactive mode. Then type run to start the program.
Inside gdb: `break main` (set breakpoint), `run` (start), `next` (step over), `info registers` (view CPU registers)
🔧 Analyzing a core dump (program crash):
gdb ./program core
→ Loads the crashed program and the core dump. Type bt (backtrace) to see the function call stack at the moment of crash.
Use case: Understanding why a program crashed without reproducing the issue.
⚠️ Note: Core dumps are often disabled by default. Enable with ulimit -c unlimited before running the program.

🧠 Memory Forensics (Volatility Framework)

Volatility is the industry standard for memory forensics — analyzing RAM dumps to find hidden processes, network connections, and malware artifacts.

📖 How to capture memory on Linux:
sudo dd if=/dev/mem of=memory.dump bs=1M (requires kernel support)
Or use avml (recommended): sudo ./avml memory.dump

Then analyze with Volatility 3:
vol3 -f memory.dump windows.pslist (list processes)
vol3 -f memory.dump windows.netscan (network connections)
vol3 -f memory.dump windows.malfind (detect injected code)

Installation:
git clone https://github.com/volatilityfoundation/volatility3.git
cd volatility3 && sudo python3 setup.py install
💡 Why memory forensics? Malware running in memory might never touch the hard drive. Traditional disk forensics can miss it entirely. Memory is where the truth lives.

⚡ Advanced Diagnostic One-Liners (Copy-Paste Ready)

🔍 Find all processes that are listening on network ports (including hidden ones):
sudo netstat -tulpn | grep LISTEN
sudo ss -tulpn (modern replacement)
🔍 Trace all new processes spawned on the system:
sudo bpftrace -e 'tracepoint:syscalls:sys_enter_execve { printf("%s -> %s\n", comm, str(args->filename)); }'
→ Shows every time any program executes another program.
🔍 Check for processes hiding from `ps`:
sudo cat /proc/*/cmdline | tr '\0' '\n' | grep -v "^$"
→ Directly reads the /proc filesystem. If a process hides from `ps` but appears here, you've found a rootkit.
🔍 Monitor file system changes in real-time (like `auditd`):
sudo bpftrace -e 'kprobe:security_file_open { printf("File opened: %s by PID %d\n", str(args->path->dentry->d_name.name), pid); }'
→ Shows every file being opened on the system.

📝 Practice Exercises (Advanced)

Exercise 1: Trace all system calls made by the `ls` command, but only show file-related operations
Solution:
strace -e trace=file ls
Exercise 2: Find out which function in `nginx` is consuming the most CPU (requires perf)
Solution:
sudo perf top -p $(pgrep nginx | head -1)
→ Profiles the first nginx process in real-time.
Exercise 3: Write a bpftrace one-liner that prints the PID and command name every time any program executes `cat`
Solution:
sudo bpftrace -e 'tracepoint:syscalls:sys_enter_execve /strcmp(args->filename, "/bin/cat") == 0/ { printf("PID %d executed cat\n", pid); }'
Exercise 4: Generate a core dump from a running process and analyze it (requires installing `gdb` and enabling core dumps)
Solution:
ulimit -c unlimited
sudo gcore [PID]
gdb /proc/[PID]/exe core.[PID]
→ Inside gdb: bt (backtrace), info registers, thread apply all bt

🎓 Key Takeaways (Level 5 - Advanced Mastery)

  • ✅ strace → See every system call. Essential for malware analysis and debugging.
  • ✅ perf → Find CPU bottlenecks at the function level, not just process level.
  • ✅ bpftrace / eBPF → Program the kernel safely. The future of Linux observability.
  • ✅ gdb → Debug crashes and reverse engineer binaries.
  • ✅ Volatility → Memory forensics — find malware that never touches disk.
  • ✅ Always test advanced tools in a VM first — they can impact system stability.
  • ✅ The /proc filesystem is your friend — when tools lie, read directly from /proc.
