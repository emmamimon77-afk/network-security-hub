🛡️ Linux Level 2: Security & Incident Response

Essential commands for security analysts, SOC teams, and incident responders. Learn to investigate network connections, analyze logs, detect anomalies, and respond to compromises.

💡 Prerequisite: Linux Level 1 (navigation, files, processes). If you're new, start there first!

🌐 Network Investigation (Find Active Connections)

CommandSyntaxDescriptionExample & Explanation
ss (Socket Statistics) ss [options] Modern replacement for netstat — shows network connections, listening ports, and sockets ss -tulpn
→ -t (TCP), -u (UDP), -l (listening), -p (process name), -n (numeric ports)
What you see: Every open port and which program owns it.
lsof (List Open Files) lsof [options] Lists all open files — including network connections, regular files, and devices lsof -i :80
→ Shows which process is using port 80 (web server)
lsof -p 1234
→ Shows all files opened by process with PID 1234
Use case: Find what's listening on a suspicious port.
🌐 Real-world incident: A server is slow. You suspect a backdoor.
ss -tulpn | grep ESTABLISHED → See active connections to suspicious IPs.
lsof -i @203.0.113.5 → Find which process is talking to that IP.
🔧 Understanding Pipes (`|`) — How This Command Works:
The pipe symbol `|` sends output from one command to the next. Order matters!

grep "Failed password" /var/log/auth.log
  → Extracts all failed login lines

| awk '{print $9}'
  → Keeps only the 9th field (IP address)

| sort
  → Sorts IPs alphabetically (required for `uniq` to work)

| uniq -c
  → Counts consecutive duplicates

| sort -nr
  → Sorts by number, highest first (reverse order)

| head -10
  → Shows only top 10 results

🔴 If you change the order, the command breaks: For example, `uniq -c` must receive sorted input, so `sort` MUST come before `uniq`!

📜 Log Analysis (Track Failures & Attacks)

CommandSyntaxDescriptionExample & Explanation
journalctl journalctl [options] View system logs from systemd's journal (modern Linux) journalctl -xe
→ -x (show explanations), -e (jump to end)
journalctl -u sshd --since "1 hour ago"
→ Shows SSH logs from the last hour only.
Use case: See why a service failed to start.
grep grep [pattern] [file] Search for text patterns in files or command output grep "Failed password" /var/log/auth.log
→ Finds all failed SSH login attempts.
grep -r "error" /var/log/
→ Recursively searches ALL logs for "error".
Use case: Find brute-force attempts in auth logs.
tail -f tail -f [file] Follow a log file in real-time (live monitoring) sudo tail -f /var/log/auth.log
→ Shows new log entries as they appear. Press Ctrl+C to stop.
Use case: Watch live attack attempts as they happen.
📋 Real-world incident: You suspect brute-force attack.
sudo grep "Failed password" /var/log/auth.log | awk '{print $9}' | sort | uniq -c | sort -nr | head -10
→ Shows top 10 IPs trying to break in, sorted by attempt count!

🔍 Process Forensics (Find Malicious Processes)

CommandSyntaxDescriptionExample & Explanation
ps (advanced) ps auxf Show all processes with tree view (parent-child relationships) ps aux --sort=-%cpu | head -10
→ Top 10 CPU-consuming processes.
ps aux | grep -E "nc|ncat|netcat"
→ Look for netcat (common backdoor).
Use case: Detect cryptominers using 100% CPU.
pstree pstree -p [PID] Show process tree with PIDs (visualize parent/child) pstree -p 1234
→ Shows process 1234 and all its children.
Use case: Find what spawned a suspicious child process.
⚠️ Red flag processes to investigate:
• nc, netcat, ncat → Reverse shells / backdoors
• minerd, xmrig → Cryptominers
• /tmp/ processes → Often malware (legit processes rarely run from /tmp)
• Processes with random names (e.g., asdfgh) → Suspicious

🔐 File Integrity & Hashing (Detect Tampering)

CommandSyntaxDescriptionExample & Explanation
sha256sum sha256sum [file] Generate SHA-256 hash of a file (fingerprint) sha256sum /bin/ls
→ Returns a unique hash like a1b2c3d4...
Use case: Compare against known good hash to detect if binary was replaced by rootkit.
find (with -perm) find / -perm -4000 -type f Find SUID files (potential privilege escalation) find / -perm -4000 -type f 2>/dev/null
→ Lists all SUID files (run as file owner, not user).
Use case: Unexpected SUID files are common backdoors.

🚨 Incident Response Quick Checklist (Copy-Paste Ready)

Step 1 — See who's logged in:
who -a | w | last -20 | lastb -10

Step 2 — Check for unusual processes:
ps aux --sort=-%cpu | head -10
ss -tulpn
lsof -i

Step 3 — Examine recent logs:
journalctl --since "1 hour ago"
grep "Failed password" /var/log/auth.log | tail -20

Step 4 — Check for persistence (cron jobs):
crontab -l
ls -la /etc/cron*

Step 5 — Find recently modified files:
find /etc /bin /usr/bin -mtime -1 -type f 2>/dev/null
(Files modified in last 24 hours)

📝 Practice Exercises (Test Your IR Skills)

Exercise 1: Find all listening ports and the processes that own them
Solution:
ss -tulpn
What to look for: Unexpected ports (e.g., 4444, 1337) or processes running from /tmp
Exercise 2: Find all failed SSH login attempts in the last 24 hours
Solution:
sudo grep "Failed password" /var/log/auth.log | grep "$(date --date='1 day ago' '+%b %e')"
Exercise 3: Find the top 10 IPs that failed to authenticate
Solution:
sudo grep "Failed password" /var/log/auth.log | awk '{print $9}' | sort | uniq -c | sort -nr | head -10
Explanation: awk extracts the IP field, sort/uniq counts duplicates, sort -nr orders by highest count.
Exercise 4: Monitor authentication logs in real-time for live attacks
Solution:
sudo tail -f /var/log/auth.log | grep --color=always "Failed\|Accepted"
→ Press Ctrl+C to stop.

🎓 Key Takeaways (Security & IR Mastery)

  • ✅ ss -tulpn → See all listening ports and their processes
  • ✅ lsof -i :port → Find what's using a specific port
  • ✅ grep "Failed" /var/log/auth.log → Track brute-force attempts
  • ✅ tail -f → Live log monitoring (press Ctrl+C to stop)
  • ✅ ps aux --sort=-%cpu → Detect cryptominers and resource hogs
  • ✅ sha256sum → Verify file integrity (detect backdoors)
  • ✅ Always check /var/log/auth.log — first stop for intrusion signs
