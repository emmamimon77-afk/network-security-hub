🔧 Linux Level 3: Hardening & System Administration

Secure your Linux systems like a pro. Learn firewall management, user security, permission hardening, SSH protection, and automated defense with fail2ban.

💡 Prerequisite: Linux Levels 1 & 2 (basics + incident response). These commands change system configuration — test in a VM first!

🔥 Firewall Management (Block the Bad Guys)

Option A: iptables (Legacy, Still Everywhere)

CommandSyntaxDescriptionExample & Explanation
iptables -L iptables -L -v -n List all firewall rules (verbose, numeric) sudo iptables -L -v -n
→ -L (list rules), -v (verbose, shows packet counts), -n (numeric IPs/ports)
What you see: Current firewall rules with packet/byte counters.
iptables -A iptables -A INPUT -p tcp --dport 22 -j ACCEPT Append a rule to a chain (add at end) sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
→ -A INPUT (add to INPUT chain), -p tcp (protocol), --dport 22 (destination port), -j ACCEPT (target action)
Translation: "Let SSH traffic in."
iptables -P iptables -P INPUT DROP Set default policy for a chain (what to do with unmatched packets) sudo iptables -P INPUT DROP
→ -P INPUT (set policy for INPUT chain), DROP (silently discard)
Translation: "Block everything unless a rule explicitly allows it."
🔧 Building a Basic Firewall (Step by Step):

sudo iptables -P INPUT DROP
  → Step 1: Block everything by default

sudo iptables -A INPUT -i lo -j ACCEPT
  → Step 2: Allow loopback (localhost communication)

sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
  → Step 3: Allow responses to traffic you initiated (e.g., browsing web)

sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
  → Step 4: Allow SSH (so you don't lock yourself out!)

sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
  → Step 5: Allow web traffic (if running a web server)
⚠️ Warning: If you set iptables -P INPUT DROP without allowing SSH first, you will lock yourself out of the server! Always add the SSH rule BEFORE setting the default DROP policy.

Option B: ufw (Uncomplicated Firewall) — Easier for Beginners

sudo ufw default deny incoming → Block all incoming traffic by default
sudo ufw default allow outgoing → Allow all outgoing traffic
sudo ufw allow 22/tcp → Allow SSH
sudo ufw allow 80/tcp → Allow HTTP
sudo ufw allow 443/tcp → Allow HTTPS
sudo ufw enable → Activate the firewall
sudo ufw status verbose → Check your rules

👤 User Management (Who Has Access?)

CommandSyntaxDescriptionExample & Explanation
useradd sudo useradd [options] [username] Create a new user account (low-level) sudo useradd -m -s /bin/bash john
→ -m (create home directory /home/john), -s /bin/bash (set default shell)
Translation: "Create user john with a home folder and bash shell."
passwd sudo passwd [username] Set or change user password sudo passwd john
→ Prompts you to enter and confirm a password for user john.
Security note: Always set strong passwords (12+ chars, mix of cases, numbers, symbols).
usermod sudo usermod [options] [username] Modify existing user account (add to groups, change shell) sudo usermod -aG sudo john
→ -aG (append to group), sudo (the group name)
Translation: "Add john to the sudo group so he can run admin commands."
userdel sudo userdel -r [username] Delete user account (and optionally home directory) sudo userdel -r john
→ -r removes home directory and mail spool
Translation: "Delete user john and all their files."
🔐 Security Best Practices:
• Never use root directly — use `sudo` with your own account
• Only add users to `sudo` group if absolutely necessary
• Regularly audit user accounts: `cat /etc/passwd`
• Remove accounts of former employees immediately: `sudo userdel -r [username]`

🔒 Permission Hardening (Protect Files & Directories)

CommandSyntaxDescriptionExample & Explanation
chmod (numeric mode) chmod 755 script.sh Set file permissions using octal numbers chmod 700 secret.txt
→ 7 (owner: read+write+execute), 0 (group: none), 0 (others: none)
Translation: "Only the owner can access this file."

Permission numbers: 4=read, 2=write, 1=execute. Add them up.
• 7 = 4+2+1 (read, write, execute)
• 6 = 4+2 (read, write)
• 5 = 4+1 (read, execute)
• 0 = no permissions
chmod (symbolic mode) chmod u+x script.sh Add or remove permissions using letters chmod u+x script.sh
→ u (user/owner), +x (add execute permission)
Translation: "Make this script executable by its owner."

chmod go-rwx secret.txt
→ go (group and others), -rwx (remove read, write, execute)
Translation: "Lock down this file from everyone except me."
chown sudo chown [user]:[group] [file] Change file owner and group sudo chown www-data:www-data /var/www/index.html
→ Changes owner to www-data (web server user)
Translation: "Give the web server ownership of this file."
⚠️ Critical Security Rule: Never set `chmod 777` on any file (gives everyone read/write/execute). This is a massive security hole. Use `700`, `750`, or `755` instead.

🛡️ SSH Hardening (Protect Remote Access)

Step 1: Edit SSH configuration file:
sudo nano /etc/ssh/sshd_config

Step 2: Make these critical changes:
PermitRootLogin no → Disable root login (attackers target root!)
PasswordAuthentication no → Disable password login, force SSH keys only
Port 2222 → Change default port 22 to something else (reduces bots)
AllowUsers alice bob → Only allow specific users to SSH in
MaxAuthTries 3 → Limit failed attempts to 3 (stops brute force)

Step 3: Restart SSH service:
sudo systemctl restart sshd
⚠️ Warning: Keep an open SSH session while testing changes, so you don't lock yourself out!
🔑 Setting up SSH keys (more secure than passwords):
On your client machine: ssh-keygen -t ed25519
Copy to server: ssh-copy-id user@server
Now test: ssh user@server (should work without password)

🤖 fail2ban — Automatic Brute-Force Protection

fail2ban monitors logs and automatically bans IPs that fail login too many times.

Step 1: Install fail2ban:
sudo apt install fail2ban -y

Step 2: Create local configuration:
sudo nano /etc/fail2ban/jail.local

Step 3: Add SSH protection:
[sshd]
enabled = true
port = ssh
maxretry = 3
bantime = 3600
findtime = 600
→ maxretry 3 (ban after 3 failures), bantime 3600 (ban for 1 hour), findtime 600 (failures must be within 10 minutes)

Step 4: Start and enable:
sudo systemctl enable fail2ban --now

Step 5: Check banned IPs:
sudo fail2ban-client status sshd
🌐 Real-world impact: After enabling fail2ban on a public server, you'll see thousands of banned IPs within a week. It stops 99% of automated attacks before they reach your authentication system.

📋 System Hardening Checklist (Quick Reference)

✅ After installing a new server, do these in order:

1️⃣ Update everything: sudo apt update && sudo apt upgrade -y

2️⃣ Create user & disable root SSH:
sudo useradd -m -s /bin/bash myuser
sudo passwd myuser
sudo usermod -aG sudo myuser
sudo nano /etc/ssh/sshd_config → set PermitRootLogin no
sudo systemctl restart sshd

3️⃣ Set up firewall: sudo ufw allow 22/tcp && sudo ufw enable

4️⃣ Install fail2ban: sudo apt install fail2ban -y && sudo systemctl enable fail2ban --now

5️⃣ Enable automatic security updates:
sudo apt install unattended-upgrades -y
sudo dpkg-reconfigure --priority=low unattended-upgrades

📝 Practice Exercises

Exercise 1: Create a new user, set a password, and add them to the sudo group
Solution:
sudo useradd -m -s /bin/bash student
sudo passwd student
sudo usermod -aG sudo student
Exercise 2: Set up a basic firewall that allows SSH, HTTP, and HTTPS, but blocks everything else
Solution (ufw):
sudo ufw default deny incoming
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
Exercise 3: Make a script executable by its owner only, and give ownership to another user
Solution:
chmod 700 my_script.sh
sudo chown student:student my_script.sh

🎓 Key Takeaways (Hardening & Administration)

  • ✅ Default deny + allow specific is the firewall golden rule
  • ✅ Never disable SSH before you've tested — keep a backup connection open
  • ✅ Use `ufw` for simplicity, `iptables` for granular control
  • ✅ Never `chmod 777` — use `700`, `750`, or `755` instead
  • ✅ Disable root SSH and use `sudo` — basic but effective
  • ✅ fail2ban blocks 99% of automated attacks — always install it
  • ✅ Test hardening changes in a VM first before production
